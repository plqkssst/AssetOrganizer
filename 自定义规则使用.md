# AssetOrganizer v2 - 自定义规则（Custom Rules）使用指南

## 1. 什么是自定义规则

自定义规则用于处理插件内置 62 种资产类型之外的**自定义 C++ 或 Blueprint 资产类**。例如：

- 自己编写的 C++ DataAsset 子类（如 `UMyGameDataAsset`）
- 第三方插件引入的特殊资产类型
- 项目特有的 Blueprint 派生类

## 2. 数据结构

每条自定义规则由 `FCustomAssetRule` 结构体定义：

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `ClassName` | FString | 资产的类名（C++ 类名或 Blueprint 前缀） | `"MyGameDataAsset"` |
| `Prefix` | FString | 命名前缀（仅用于 UI 展示） | `"MDA_"` |
| `TargetPath` | FString | 整理时的目标文件夹路径 | `"/Game/Data/MyGame"` |
| `CategoryName` | FString | UI 中显示的类别名称 | `"Custom"` |
| `bEnabled` | bool | 是否启用该规则 | `true` |

### ClassName 的下拉列表选择

**插件面板**和 **Project Settings** 中的 `ClassName` 字段均已改为下拉列表，选项自动收集自：
- 引擎 `AssetTools` 模块注册的所有资产类型（Content Browser 可创建的类型）
- `AssetRegistry` 扫描 `/Game` 路径发现的项目中实际存在的资产类
- 已有的自定义规则类名

系统会自动去重排序，无需手动输入类名。如果某类同时属于内置 62 种类型，添加时会自动复用其 Prefix 和 TargetPath。

### 如何确定 ClassName（手动方式）

在 Content Browser 中右键资产 → **Copy Reference**，粘贴后取最后一个 `.` 之前的部分：

```
/Game/SomeFolder/MyAsset.MyAsset  ← 这是 ObjectPath
              ↑
        在 AssetData 中，AssetClass 字段保存的是这个资产的实际类名
```

更准确的方式：在 Content Browser 中选中资产，看详情面板的 **Class** 字段。例如：
- C++ 类 `UMyGameDataAsset` → ClassName 填 `"MyGameDataAsset"`
- Blueprint 继承自 `MyGameDataAsset` → ClassName 同样填 `"MyGameDataAsset"`（引擎内部 Blueprint 资产的 AssetClass 是其父类）

## 3. 操作步骤

### 3.1 在插件窗口中添加规则

1. 打开编辑器 → **Window** → **Asset Organizer**
2. 在主列表下方找到 **Custom Rules** 区域
3. 点击 **Class** 下拉列表，选择资产类名
   - 下拉列表包含引擎自带的所有资产类型和项目中实际存在的资产类
   - 若该类属于内置 62 种类型，系统会自动填充推荐的 Prefix 和 TargetPath
4. 点击 **+ Add Rule** 按钮添加规则
5. 新规则立即显示在列表中，同时同步到 Project Settings
6. 点击 **Remove** 删除不需要的规则
7. 点击 **Refresh** 手动从 Project Settings 同步最新规则状态

### 3.2 编辑规则（Project Settings）

**Edit → Project Settings → Plugins → Asset Organizer → Custom Rules**

1. 点击已有规则旁的编辑按钮展开字段：
   - **Class Name**: 下拉列表选择资产类名（选项与插件面板一致）
   - **Prefix**: 设置命名前缀（可选，仅展示用）
   - **Target Path**: 设置目标文件夹（如 `/Game/Data/MyType`）
   - **Category Name**: 设置 UI 类别名称（可选）
   - **Enabled**: 勾选/取消以启用或禁用
2. 修改实时自动保存到 `Config/AssetOrganizer.ini`
3. 回到 Asset Organizer 窗口，规则已自动同步（或点击 **Refresh** 手动刷新）

### 3.3 执行整理

自定义规则资产会像内置类型一样参与整理流程：

1. 确保自定义规则已启用（`bEnabled = true`）
2. 点击 **一键整理** 或 **预览**
3. 匹配自定义规则的资产会被识别、归类并移动到 `TargetPath`

## 4. 自定义规则的生效流程

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 扫描资产                                              │
│   AssetRegistry 扫描 /Game 下的所有资产                       │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 2: 类型判定 (DetermineAssetType)                         │
│   优先检查 CustomRules：                                      │
│   - 遍历 Settings->CustomRules                                │
│   - 若 AssetClass == Rule.ClassName 且 Rule.bEnabled          │
│   - 返回 EAssetOrganizeType::Custom                           │
│                                                               │
│   若未匹配任何自定义规则，则检查内置 62 种类型                │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 3: 分组 (GroupAssetsByType)                              │
│   所有匹配自定义规则的资产归入同一个 Custom 组                │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 4: 确定目标路径                                          │
│   构建 Config.TypeToFolderMap：                               │
│   - 内置类型的目标路径来自 UI 中设置的 TargetPath             │
│   - Custom 类型的目标路径来自第一条启用的规则的 TargetPath    │
│                                                               │
│   调用 Config.GetTargetPath(Custom, DefaultPath)：            │
│   - 先查 TypeToFolderMap，找到则返回 Rule.TargetPath        │
│   - 否则返回默认路径 /Game/Other                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 5: 移动资产                                              │
│   将 Custom 组中的资产移动到 Rule.TargetPath                  │
│   完成后自动修复引用（如果启用了 bFixReferences）             │
└─────────────────────────────────────────────────────────────┘
```

## 5. 实际示例

### 示例 1：自定义 DataAsset

假设项目中有一个 C++ 类：

```cpp
UCLASS()
class MYGAME_API UMyGameConfigAsset : public UDataAsset
{
    GENERATED_BODY()
};
```

在 Content Browser 中创建一个基于 `MyGameConfigAsset` 的 Blueprint 资产，命名为 `BP_PlayerConfig`。

**自定义规则设置**：

| 字段 | 值 |
|------|-----|
| ClassName | `MyGameConfigAsset` |
| Prefix | `CFG_` |
| TargetPath | `/Game/Data/Configs` |
| CategoryName | `Data` |
| bEnabled | ✅ |

**整理结果**：
- 所有 `MyGameConfigAsset` 类型的资产（包括 Blueprint 派生）会被移动到 `/Game/Data/Configs`
- 命名自动应用前缀 `CFG_`（如果启用了重命名功能）

### 示例 2：第三方插件资产

假设使用了某个插件，引入了一种新资产类型 `UMyPluginTexture`。

**自定义规则设置**：

| 字段 | 值 |
|------|-----|
| ClassName | `MyPluginTexture` |
| Prefix | `MPT_` |
| TargetPath | `/Game/Textures/Plugin` |
| CategoryName | `Textures` |
| bEnabled | ✅ |

## 6. 已知限制

### 6.1 多条自定义规则共享同一个目标路径

**当前版本中，所有自定义规则资产会被移动到同一个目标路径。**

原因：内部设计上，所有自定义规则资产都被归为 `EAssetOrganizeType::Custom` 这一个类型组。`TypeToFolderMap` 中 `Custom` 键只能对应一个路径值。实际使用的是**第一条启用的自定义规则**的 `TargetPath`。

**影响**：
- ✅ 如果只定义了一条自定义规则 → 正常工作
- ⚠️ 如果定义了多条不同 `TargetPath` 的规则 → 所有自定义规则资产都会被移到第一条规则的路径

**解决方案**：
- 当前版本：将所有同类自定义资产放在同一个文件夹下
- 后续版本：计划支持按具体 ClassName 细分目标路径

### 6.2 多条自定义规则共享同一个目标路径

**当前版本中，所有自定义规则资产会被移动到同一个目标路径。**

原因：内部设计上，所有自定义规则资产都被归为 `EAssetOrganizeType::Custom` 这一个类型组。`TypeToFolderMap` 中 `Custom` 键只能对应一个路径值。实际使用的是**第一条启用的自定义规则**的 `TargetPath`。

**影响**：
- ✅ 如果只定义了一条自定义规则 → 正常工作
- ⚠️ 如果定义了多条不同 `TargetPath` 的规则 → 所有自定义规则资产都会被移到第一条规则的路径

**解决方案**：
- 当前版本：将所有同类自定义资产放在同一个文件夹下
- 后续版本：计划支持按具体 ClassName 细分目标路径

### 6.3 规则匹配优先级

自定义规则的检查**优先于**内置类型。如果某资产的类名同时匹配自定义规则和某个内置类型，优先按自定义规则处理。

例如：如果为 `"DataTable"` 定义了自定义规则，那么 DataTable 资产会被当作 Custom 类型而不是内置的 DataTable 类型。

## 7. 配置文件位置

自定义规则保存在项目配置文件中：

```
Test1/Config/AssetOrganizer.ini
```

示例内容：

```ini
[/Script/AssetOrganizer.AssetOrganizerSettings]
CustomRules=(ClassName="MyGameConfigAsset",Prefix="CFG_",TargetPath="/Game/Data/Configs",CategoryName="Data",bEnabled=True)
CustomRules=(ClassName="MyPluginTexture",Prefix="MPT_",TargetPath="/Game/Textures/Plugin",CategoryName="Textures",bEnabled=False)
```

可以手动编辑此文件来批量添加规则，修改后重启编辑器生效。

## 8. 常见问题

### Q1: 添加了规则但资产没有被识别？

- 确认 `ClassName` 选择正确（区分大小写）
- 确认 `bEnabled` 为 `true`
- 在 Content Browser 中选中资产，检查 Details 面板中的 **Class** 字段
- 点击 Asset Organizer 窗口中的 **Refresh** 按钮刷新规则列表

### Q2: 自定义规则资产被移动到了 /Game/Other？

- 确认规则已启用
- 确认规则的 `TargetPath` 不为空
- 检查是否有其他规则也启用了（当前版本只使用第一条启用的规则路径）
- 在插件面板或 Project Settings 中检查规则状态是否同步

### Q3: 如何验证规则是否生效？

使用 **预览** 功能：
1. 点击 **预览** 按钮
2. 查看 Preview 列表中是否有匹配自定义规则的资产
3. 检查 "To" 列是否显示正确的目标路径

### Q4: 插件面板和 Project Settings 的规则不同步？

- 双方已配置双向自动同步：任一方修改后，另一方会自动刷新
- 若自动同步未及时生效，点击插件面板中的 **Refresh** 按钮手动同步
- 确认 `UAssetOrganizerSettings` 继承自 `UDeveloperSettings`（v2.0+ 版本已修复）

### Q5: Blueprint 资产和 C++ 资产如何区分？

在 UE4.27 中，Blueprint 资产的 `AssetClass` 是其父类名，而不是 `"Blueprint"`。例如：
- 继承自 `MyGameConfigAsset` 的 Blueprint → AssetClass = `"MyGameConfigAsset"`
- 普通 Blueprint（继承 AActor）→ AssetClass = `"Blueprint"`

所以自定义规则的 `ClassName` 应该填写父类名。

## 9. 完整数据流

```
Project Settings (CustomRules[])
         ↓ 双向同步
UAssetOrganizerSettings::CustomRules (config)
         ↓ OnCustomRulesChanged 委托广播
SAssetOrganizerWidget::Construct()
  → RebuildCustomRulesList() (显示在 UI)
  → RefreshClassSelectorOptions() (更新下拉列表)
         ↓
StartAsyncOrganize()
  → Config.TypeToFolderMap[Custom] = Rule.TargetPath
         ↓
FAssetOrganizerExecutor::OrganizeAssetsAsync()
  → DetermineAssetType(Asset, &CustomRules)
    → 匹配？返回 Custom : 检查内置类型
  → GroupAssetsByType() → Custom 组
  → GetTargetFolder(Custom, Config)
    → Config.GetTargetPath(Custom, "/Game/Other")
      → TypeToFolderMap.Find(Custom) → Rule.TargetPath
  → MoveAsset() → 移动到 Rule.TargetPath
```

**双向同步机制**：
- **Project Settings → 插件面板**：Project Settings 中修改 `CustomRules` 后，`PostEditChangeProperty` 自动保存配置并广播 `OnCustomRulesChanged` 委托；Widget 订阅该委托，自动重建列表和刷新下拉选项
- **插件面板 → Project Settings**：插件面板添加/删除规则后，调用 `Settings->PostEditChange()` 触发 Details 面板刷新，同时 `SaveConfig()` 持久化到 `AssetOrganizer.ini`
- **手动同步**：点击插件面板的 **Refresh** 按钮可强制从 Settings 重新加载并刷新 UI
